# 实验一 2.1 免费资源下模型推理能力分析最终报告

## 结论先行

**单张 T4 上的 4bit / 8bit 推理**
因为 4bit 和 8bit 都能完成 `加载 -> 生成 -> 解码 -> official eval`，

parser / 输出格式不稳定
**SmartTest 输出都没有生成规范的唯一 `<answer>...</answer>`

fp16 单卡在 T4 上加载阶段 OOM。


最终判断：
免费 Kaggle T4 资源可以完成小样本推理链路验证，但只能作为工程可行性验证；
量化加载虽然解决了显存问题，但生成速度慢、输出格式不稳定、official 指标较差，不能据此证明模型能力复现成功。

## 4. 延迟是多少？

**简洁回答：**  
SmartTest 2048 token 设置下，4bit 很慢，平均约 **3507.756 秒/样本**；8bit 平均约 **1140.646 秒/样本**，但它实际平均生成 token 更少。

**详细回答：**

| 配置          | avg latency | max latency | avg actual new tokens | tokens/s |
| ----------- | ----------: | ----------: | --------------------: | -------: |
| 4bit single |  3507.756 s |  4060.895 s |                2048.0 |    0.599 |
| 8bit single |  1140.646 s |  1242.483 s |                1068.5 |    0.952 |

## 5. 生成质量如何？`smarttest_0` 和 `smarttest` 怎么对比？

**简洁回答：**  
生成质量不稳定。
经常没有规范 `<answer>...</answer>`，forecasting 会重复或打满 token，上游格式控制提示词也没有稳定改善 parser 成功率。



## 6. 结论是什么？

**简洁回答：**  
免费 Kaggle T4 资源可以完成小样本推理链路验证，但当前只能证明“跑得通”，不能证明“生成质量复现成功”。4bit / 8bit 是可行替代方案，但量化加载和低资源环境会影响速度、输出收束和格式稳定性。

**详细回答：**
- 4bit 单卡 T4：能加载、能推理、显存余量足，但速度极慢，输出不稳定。
- 8bit 单卡 T4：能加载、能推理，显存峰值更高，速度仍慢，输出质量仍不可接受。
- fp16 单卡 T4：加载阶段 OOM，不适合作为免费 T4 单卡方案。

主要限制来自三层：
1. 环境限制：Kaggle T4 上 `flash_attn` 编译和版本匹配成本高，所以实验选择不安装 `flash_attn`，改用 `sdpa` 或 `eager`。证据见 `00_scripts介绍.md:31-32`。
2. 加载与显存限制：单卡 fp16 在 14.56 GiB T4 上 OOM；4bit / 8bit 依靠 bitsandbytes 量化才能加载。量化配置由 `run_experiment1_new_version.py:350-360` 的 `BitsAndBytesConfig` 控制。
3. 输出与 parser 限制：脚本生成后保存 `decoded_text`，strict diagnostic 要求唯一 `<answer>`，official eval 则更宽松地从 `response` 字段解析。这两个口径不同，导致“生成成功”“strict parse 成功”“official eval 有覆盖率”不能混为一谈。相关代码分别在 `run_experiment1_new_version.py:686-688`, `:737-787`, `:858-878` 和 `evaluation/evaluate_qa.py:88-132`。
