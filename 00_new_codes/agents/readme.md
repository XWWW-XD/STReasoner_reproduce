### Codex 报告模板与书写规则

用途：每次 Codex 完成环境修复、代码修改、实验运行、脚本分析后，都需要在 `reports/` 目录生成一份简洁报告，方便之后回看和汇报。

建议文件名：

```text
reports/YYYYMMDD_task_name_report.md
```

例如：

```text
reports/20260528_codex_ui_reconnect_fix_report.md
```

### 报告写作原则

报告要服务于“之后能快速看懂发生了什么”，不是写流水账。

要求：

* 先写结论，再写过程；
* 区分已确认事实、合理推测、未解决问题；
* 不要堆太多终端输出，只保留关键命令和关键结果；
* 不要泄露 token、密钥、账号凭证、完整路径中的敏感信息；
* 修改过的文件必须列出；
* 如果没有真正修复，要明确写“未解决”，不要假装完成；
* 如果需要用户手动操作，要写清楚下一步命令或动作。

### 标准报告模板

````md
### 任务目标

简要说明这次任务要解决什么问题。

例如：

- 修复远程 VS Code 中 Codex UI 一直 reconnect 的问题；
- 检查 Stage 2.2 推理脚本是否能正常读取数据并生成结果；
- 分析某个实验输出文件的含义。

### 当前结论

用 2-5 条写最重要结论。

格式建议：

- 已确认：
- 可能原因：
- 当前状态：
- 是否完成：

示例：

- 已确认：终端中的 Codex CLI 可以正常启动。
- 已确认：右侧 VS Code Codex UI 报错中出现 `token_invalidated / status 401`。
- 可能原因：右侧 UI 使用的 Codex app-server / MCP 认证状态失效。
- 当前状态：已重新检查认证和配置，但仍需要用户重载 VS Code 窗口后验证。

### 已知现象

记录任务开始前观察到的问题。

包括：

- 报错信息；
- 哪些功能可用；
- 哪些功能不可用；
- 问题出现的条件。

示例：

```text
Codex CLI 可以在远程终端中启动；
VS Code 右侧 Codex UI 可以打开，但一直 reconnect；
日志中出现 token_invalidated / status 401。
````

### 执行过程

按时间顺序简要记录做过什么，不要写太细。

建议格式：

1. 检查了什么；
2. 发现了什么；
3. 修改了什么；
4. 重新测试了什么。

示例：

* 检查 `~/.codex/config.toml`，确认代理配置存在；
* 检查 `codex login status`，确认 CLI 当前登录状态；
* 检查代理环境变量，确认 `HTTPS_PROXY` 指向远程代理端口；
* 未打印或泄露 `auth.json` 内容。

### 修改内容

列出所有修改过的文件。

格式：

```text
修改文件：
- path/to/file1：修改原因
- path/to/file2：修改原因

新增文件：
- path/to/file3：新增原因

未修改项目代码文件。
```

如果没有修改文件，也要写：

```text
本次未修改任何项目代码文件。
```

### 关键命令与结果

只保留关键命令，不要复制大量无关输出。

格式：

```bash
命令
```

结果：

```text
关键输出
```

示例：

```bash
codex login status
```

结果：

```text
Logged in using ChatGPT
```

### 问题原因分析

分三层写：

```md
已确认原因：
- ...

合理推测：
- ...

尚不确定：
- ...
```

不要把推测写成事实。

### 当前状态

说明现在到底到了哪一步。

可选状态：

```text
已解决
部分解决
未解决
需要用户手动确认
需要进一步排查
```

并写清楚判断依据。

### 下一步操作

如果还需要用户操作，写成可直接执行的步骤。

示例：

```text
1. 在 VS Code 中执行 Developer: Reload Window；
2. 重新打开右侧 Codex UI；
3. 如果仍然 reconnect，重新记录最新日志。
```

如果不需要下一步，写：

```text
暂无额外操作。
```

### 简短总结

用 3-5 句话总结：

* 这次解决了什么；
* 没解决什么；
* 下次遇到同类问题应该先看哪里。

```

### 报告质量检查

提交报告前检查：

- 是否先写了结论；
- 是否说明了任务是否完成；
- 是否列出修改文件；
- 是否区分事实和推测；
- 是否写了下一步；
- 是否避免泄露 token / key / auth.json 内容；
- 是否避免复制大量无关日志。

### 给 Codex 的固定要求

每次任务结束后，请在 `reports/` 目录生成一份新的 md 报告。

报告必须包括：

- 任务目标；
- 当前结论；
- 已知现象；
- 执行过程；
- 修改内容；
- 关键命令与结果；
- 问题原因分析；
- 当前状态；
- 下一步操作；
- 简短总结。

报告要简洁、可复盘、可用于之后汇报。不要写成流水账，也不要假装解决未解决的问题。
```
